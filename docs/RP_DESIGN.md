# 可控 Roleplay 设计书（v1，待确认）

> 定位：**为写作服务的受控角色扮演**——多座席（KP + 角色）各持独立上下文，
> 世界上下文负责裁决与事件分发；作者掌握每个座席的最终发言权；
> RP 产出结构化记录（transcript），第二阶段改写成小说（novelize，复用卷一管线）。
> 本文以写作方法论为第一性原理反推系统设计。

---

## 一、七条写作方法论 → 七个系统机制

| 写作方法论 | 系统机制 |
|---|---|
| **场景是叙事的基本单位**（Scene: 目标→冲突→转折；不是"回合"也不是"天"） | 自由发言流被**场景容器**包裹：KP 开场（时间/地点/在场者），场内自由发言，KP 或世界**关闭场景=时间推进+结算**。这正是"KP 和世界推动时间流逝"的形式化 |
| **视点纪律**（每场景一个 POV；作者写 A 时要"忘掉"B 知道的事） | 座席上下文隔离不是工程洁癖，而是 POV 纪律的机器强制。novelize 选 A 视点时，用的就是 A 座席的所知流+内心层，**结构上不可能全知泄漏** |
| **草稿与修订分离**（先让故事发生，再写成文） | RP 阶段不追求文笔，追求**事件密度与心理真实**；文笔全部留给 novelize。因此 RP 的发言格式是三层结构而非漂亮散文（见 §二） |
| **人物 = 人设 × 动机 × 信息状态**（作者随时知道"他此刻知道什么/想要什么/怕什么"） | 角色座席的核心不是聊天记录，而是**状态卡**：长期人设 + 当前目标 + 情绪 + 关系 + 已知信息摘要。场景结算时状态卡打 delta（AI 自结算、人过目）——这就是人物弧线的机器载体 |
| **伏笔账本**（埋了什么、何时回收，成熟作者用卡片管理） | 世界上下文维护**线头账本**（open threads）：伏笔、悬念、未兑现的承诺。KP 面板可见；KP 的 AI 在建议下一场景时引用（"缇月民俗已埋三场未动"） |
| **节奏靠场景-间奏交替**（张弛曲线） | 场景之外支持**间奏**（sequel）：单角色独处的"反应→两难→决定"小场景，为 novelize 储备心理素材。KP 面板给节奏仪表：近期冲突强度、距下个天象锚点的天数（"日食还有 9 天"） |
| **修订权**（作者可以划掉任何一句重来） | 任何发言可编辑/删除/重演。未结算场景内随意改（还没有下游）；已结算的改动走 retcon changeset，世界上下文标记并撤回受影响的下游分发 |

## 二、运行模型：场景舞台（按你已定的 1/2/3 展开）

```
┌─ KP 座席（作者默认位）──────────────────────────────┐
│ 全知视图 | 线头账本 | 节奏仪表 | 世界查询(route/sky/…) │
└──────────────┬───────────────────────────────────────┘
               │ ①开场包 / ⑤关闭场景
┌──────────────▼───────────────────────────────────────┐
│ 世界上下文 = 轻裁决(场内,即时) + 重结算(关场,批量)      │
│   裁决工具：route/viewshed/news_arrival/sky/describe   │
└───┬──────────────────────────────────────────────▲───┘
    │ ②按视角分发开场               ③行动/台词/内心 │
┌───▼──────────┐  ┌──────────────┐  ┌──────────────┴─┐
│ 角色座席 A    │  │ 角色座席 B    │  │ 角色座席 C(自动)│
│ 状态卡+所知流 │  │ 状态卡+所知流 │  │ 状态卡+所知流   │
└──────────────┘  └──────────────┘  └────────────────┘
```

- **① 开场**：KP 定时间/地点/在场者/冲突提示。世界把开场包按各角色的信息
  状态过滤后分发（同一场景，A 看到的和 B 看到的开场词不同）。
- **② 场内自由发言（实时感的来源）**：任何座席随时发言。世界对每条做
  **轻裁决**（快模型 + 查询，秒级）：可行→即时广播给在场者；涉及事实→
  附注（"此处到城门还有 6 里，天黑前赶得到"）；不可能→打回附理由
  （"骑马最快 7 天，'三天赶到王都'不成立"）。**一致性由此从事后校验
  升格为事前物理**——不可能的事进不了记录。
- **③ 三层发言格式**（角色座席的输出规格，也是 novelize 的原料分级）：
  - `行动`：可被他人观察 → 广播
  - `台词`：可被听见 → 广播
  - `内心`：**只进本座席所知流与 transcript**，其他角色永远看不到。
    小说正文里"细微的想法"全部来自这一层——这是 RP 记录比事实包稠密
    得多的根本原因，也是"人物模拟要用大模型"的着力点
- **⑤ 关闭场景（重结算）**：时钟推进（场景耗时由 KP 定）；正典化（本场
  事件写成 staged changesets，作者 merge）；**场外分发**（news_arrival
  延迟投递给不在场角色："第 X 天，你在佩维克听说了……"）；状态卡 delta；
  线头账本更新。

作者主权的实现：作者=所有座席的 merge 权持有人。每个座席可设自动化档位
（`manual` 逐条审 / `assist` AI 起草人放行 / `auto` 自动通过，事后可 retcon）
——群演 NPC 设 auto，主角和 KP 设 assist，这就是"KP 一路决定+AI 自动推算"
的可调实现。

## 三、AI 池（多 vendor）

**方法论对应：编辑部不同工种用不同的笔。**

| 工种 | 任务特征 | 建议档位 |
|---|---|---|
| 人物演员（主要角色） | 人设约束下的深层心理推理，内心层质量决定成书上限 | **最大杯**（Claude/GPT/Gemini 旗舰，按人设气质选型：不同角色可用不同 vendor，风格差异反而有利于人物区分度） |
| 人物演员（NPC/群演） | 人设一致即可 | 中杯（DeepSeek/Kimi/GLM/mini 级） |
| KP 助手 | 结构感、节奏、线头管理 | 大杯 + 高推理档 |
| 世界轻裁决 | 规则判断+查询解释，要快 | **小快杯**（秒级响应撑起"实时感"） |
| novelize 主笔 | 文笔 | 大杯高推理（卷一用 sol high 已验证） |
| 美术 | 生图 | gpt-image-2 / nano banana pro |

工程抽象 `llmpool.py`：
- 事实：九家里除 Anthropic（messages API）和 Gemini（generateContent）外，
  **OpenAI/Grok/DeepSeek/Kimi/GLM/OpenRouter/任意自定义 URL 全是
  OpenAI-compatible**。所以一个 openai 协议客户端 + 两个薄适配器 = 全覆盖。
- 配置 `ai_pool.json`（不入库，模板入库）：

## 四、渲染参考包（Scene Reference Pack）：物理引擎的独有红利

**核心论点：我们的场景不只"有设定"，而是可渲染、可查询的连续物理场。**
SillyTavern 式的文字世界书（world info/lorebook）是"设定的缓存"——关键词
触发的静态片段：条目之间没有空间连续性，没有时间演化，回答不了"角色站在
这里、此刻、朝北看，看见什么"。我们的世界是"设定的函数"：任意（地点,
时刻, 视角）都能算出一致的答案，且所有座席共享同一份物理真相。

每次开场，世界上下文自动生成**渲染参考包**，四件套全部来自现有管线：

| 渲染物 | 来源 | 形态 |
|---|---|---|
| **天际线剖面** | viewshed 升级版：从观察点沿 36-72 方位角采样高程（全局 raster 或 refine 瓦片），得仰角剖面 | 折线数据 + 轮廓图（"西北 12 公里一道 600 米山脊压住地平线"） |
| **俯视小地图** | 高程/水系/植被 raster 切片 + hypsometric/biome 着色 + 聚落道路标注 | PNG + 图例 |
| **星穹图** | astro sky()：太阳/伴日/双月（相位）/亮星的 alt-az + 星等 + **黑体真实颜色** | 半球投影 PNG + 结构化列表 |
| **光照与体感侧写** | 太阳高度角（时辰色温）、moonlight 照度、伴日贡献、气温海拔修正、降水/风带 | 纯数值 + 一句话侧写 |

**两条消费路径**：

1. **渲染引导生图（render-guided generation）**：天际线轮廓/小地图作为
   参考图或构图约束喂给 gpt-image-2 / nano banana pro——生成场景图里
   远山的形状就是**真实地形的形状**，天上双月的位置相位就是**历表的位置
   相位**。生图从"凭 prompt 想象"变成"给物理上色"。
2. **机器视野报告进 LLM**：参考包文字化后注入开场包与角色上下文
   （"河在东南 2 公里，河谷下切约 40 米，可见对岸渡口灯火"），多模态
   座席可直接喂渲染图。角色的环境描写精度立刻上一个数量级，且各座席
   引用的是同一份渲染，**描写不会互相矛盾**。

参考包同时写入 transcript 的场景头（`refpack` 字段），novelize 阶段主笔
拿到的环境素材与 RP 时各座席看到的完全同源。

## 五、生图：视觉圣经

**方法论对应：影视/IP 工业的概念设定集**——不是装饰，是设定一致性的工具。

- **三类图**：人设定妆图（角色卡建立时生成，之后作为一致性参考）、
  场景氛围图（KP 开场时一键，以 §四渲染参考包为构图底）、关键道具/
  事件插图。
- **世界事实自动入 prompt + 渲染参考图入构图（独有优势）**：提示词由
  开场包+模拟查询拼装（双日、当夜月相、生态降水、时辰光照），构图由
  天际线剖面/小地图约束。**图不会画错天空，也不会画错山的形状**。
- **一致性**：角色卡存"外貌正典描述 + 定妆图路径"；后续生成走参考图/
  image-edit 模式（gpt-image-2 与 nano banana pro 均支持）。
- 工程 `imagegen.py`：后端 gpt-image-2（OpenAI images API）+
  nano banana pro（Gemini generateContent 图输出）；图片存
  `worlds/<world>/assets/`（不入 git），元数据（prompt/参考图/座席/场景）
  入 `.weworld`；`/rp/image` 端点 + transcript 里插图引用。

## 六、transcript 规格（novelize 的合同）

JSONL，三种记录：
```
{"type":"scene", "id":7, "day":143.6, "place":[lon,lat], "cast":["凯尔","赛芮"],
 "sky":{...当夜月相/天象...}, "refpack":{"skyline":..., "minimap":...,
 "lighting":...}, "opening":"...", "threads_touched":[3,5]}
{"type":"say", "scene":7, "seat":"赛芮", "layer":"inner",   // action|dialogue|inner
 "text":"...", "arbiter_note":null, "edited_by_author":false}
{"type":"settle", "scene":7, "elapsed_days":0.3, "canon_events":[...changesets...],
 "distributed":[{"to":"伊莎","arrives_day":147.8,"summary":"..."}],
 "statecard_deltas":{...}, "threads":{...}}
```
novelize = 选 POV → 按场景切片 → 该 POV 的所知流+内心层做素材 →
卷一管线逐章生成（事实包换成场景切片），行程/天象校验原样复用。

## 七、落地里程碑（沿用后端先行原则）

- **RP-1 内核（纯 pytest，最先做）**：`llmpool.py`（多 vendor + 用量记账）+
  `rpkit.py`（座席/状态卡/场景容器/三层发言/轻裁决/结算分发/线头账本/
  transcript）。AI 可 mock，全部逻辑离线可测。
- **RP-2 渲染参考 + 生图**：`refpack.py`（天际线剖面/小地图/星穹图/光照
  侧写——全部纯计算可 pytest）+ `imagegen.py` 两后端（渲染引导生成）；
  开场包自动附机器视野报告。
- **RP-3 多面板 Web**：KP 面板（账本/仪表/查询）+ 角色面板（状态卡/所知流/
  三层输入）+ 导演视图（全景流）；Playwright 截图当验收。
- **RP-4 novelize 打通**：transcript→分章→生成→校验，复用 vol1_generate。

## 八、遗留开放问题（不阻塞 RP-1）

1. 多人协作（多作者各控一角）要不要进 RP-3 的范围？
2. 轻裁决的打回是否允许角色座席"坚持"（override 并标记为反物理的艺术处理）？
3. 状态卡的情绪/关系是否量表化（数值）还是纯文本？（建议先纯文本，量表是伪精度）
